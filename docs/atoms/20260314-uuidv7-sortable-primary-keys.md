# UUIDv7 gives sortable primary keys with embedded creation timestamps

Amos uses UUIDv7 as the primary key type across all resources. This is a deliberate choice over both auto-incrementing integers and UUIDv4 (random UUIDs).

UUIDv7 embeds a Unix timestamp in the most significant bits of the UUID. This gives two properties that matter for a PMS:

**Time-sortable by default**: Records can be sorted by primary key to get chronological order without needing a separate `created_at` column or index. For a system tracking reservations, tasks, transactions, and messages — all inherently temporal — this is a natural fit. Querying "recent tasks" or "latest transactions" can leverage the primary key index directly.

**Globally unique without coordination**: Unlike auto-increment IDs, UUIDv7s can be generated by any node without a centralized sequence. This does not matter much for a single-business system today, but it prevents ID collisions if data ever needs to be merged from different sources (e.g., syncing from a mobile app that creates records offline).

**B-tree friendly**: Unlike UUIDv4 (which is random and causes index fragmentation), UUIDv7's time-ordered nature means new records are always appended to the end of B-tree indexes. This matches the write pattern of auto-increment IDs while retaining UUID benefits.

The trade-off is storage size (16 bytes vs 4-8 bytes for integers) and human readability (UUIDs are not easy to reference verbally). For a system where IDs are primarily machine-consumed via API, this trade-off favors UUIDv7.

UUIDv7 is relatively new (RFC 9562, published 2024) and not yet universally supported. Ash and Ecto handle it well in the Elixir ecosystem, but external integrations may need conversion utilities.
