# Cross-segment flexibility between hotels and vacation rentals is an underserved market gap

The research identifies a clear opportunity: "Designing the Tasks domain to work for both hotel-style (room-level, daily operations) and vacation-rental-style (property-level, turnover-driven) workflows would address a gap in the market where most platforms serve only one segment."

**Why the gap exists**: Hotel and vacation rental operations differ in fundamental ways — task types, scheduling patterns, staffing models, quality assurance methods, and maintenance workflows all diverge. Building for both requires abstractions that are flexible enough to accommodate these differences without becoming so generic that they serve neither well.

**The convergence trend**: The hotel and vacation rental segments are slowly converging as more properties are managed by professional operators. A boutique hotel that also manages nearby vacation villas. A resort with traditional rooms and self-contained cottages. These hybrid operations need a single system that handles both workflows.

**What cross-segment flexibility requires in the Tasks domain**:
- Generic task model with type classification (not separate models per task type)
- Flexible location model (rooms, units, properties, facilities, zones)
- Configurable scheduling (daily cycle for hotels, calendar-driven for VRs)
- Dual assignment modes (employee shift assignment and contractor per-clean assignment)
- Adaptable checklists (room-type-specific for hotels, property-specific for VRs)
- Photo verification (growing in hotels, already standard in VRs)

**The competitive moat**: Standalone platforms cannot easily cross segments because they lack the PMS integration needed for the other segment's workflow. A VR platform like Breezeway would need hotel PMS integration to serve hotels. A hotel platform like Optii would need channel manager integration to serve VRs. Amos, as a PMS that serves the property directly, can build the Tasks domain to handle both without external integration dependencies.

**Risk**: Trying to serve both segments could result in serving neither well. The key is finding the right level of abstraction — flexible enough for both, but not so generic that common workflows require excessive configuration.
